# 왕과 사는 남자 인터랙티브 팬사이트 MVP 기획서
_목표: Cursor에서 바로 구현 가능한 수준의 상세 명세서_

## 0. 문서 목적
이 문서는 **영화 《왕과 사는 남자》의 여운을 현실 공동체 딜레마와 연결한 인터랙티브 웹사이트**를 만들기 위한 **실행형 기획서**다.  
목표는 단순한 캐릭터 테스트가 아니라 다음 흐름을 완성하는 것이다.

**랜딩 → 충성도 테스트 → 결과 희귀도 공개 → 분기형 게임 → 공유**

이 문서를 Cursor에 넣고, Python 중심으로 구현하도록 지시하면 된다.

# 1. 서비스 개요

## 1.1 서비스명
### 메인 서비스명
**왕과 사는 남자: 충성도 테스트**

### 서브카피
**회사, 학교, 팀플 속 나는 충신일까 생존자일까**

## 1.2 서비스 핵심 가치
이 서비스는 아래 4가지를 동시에 달성해야 한다.

1. **영화 여운 유지**  
   영화를 본 뒤 “나라면 어떤 선택을 했을까?”를 계속 생각하게 만든다.

2. **현실 공감**  
   정치 이야기가 아니라 회사, 학교, 팀 프로젝트, 친구 관계 같은 공동체 딜레마로 번역한다.

3. **바이럴성**  
   “나는 충신 82%”, “상위 11%”, “왕을 지킬 자격이 있습니다” 같은 결과가 공유되도록 만든다.

4. **체류시간 증가**  
   테스트에서 끝나지 않고 결과에 따라 다른 게임/인터랙션으로 이동하게 만든다.

# 2. 타겟 유저

## 2.1 1차 타겟
- 영화를 본 관객
- SNS에서 영화 밈/팬콘텐츠를 소비하는 사용자
- 테스트/밸런스게임/가벼운 웹게임을 좋아하는 10~30대

## 2.2 2차 타겟
- 영화를 보지는 않았지만, 후킹 문구와 결과 공유를 보고 들어온 사용자
- 커뮤니티형 바이럴 콘텐츠를 즐기는 직장인/대학생

# 3. MVP 범위

## 3.1 1차 배포에 꼭 포함할 기능
- 랜딩 페이지
- 충성도 테스트 10문항
- 결과 페이지

- 관리자 대시보드

# 4. 핵심 차별점

기존 팬사이트가 주로 하는 것은 다음이다.

- 캐릭터 테스트
- MBTI식 유형 분류
- 영화 인물 매칭

이 서비스는 다르게 가야 한다.

## 차별화 포인트
### 4.1 성격이 아니라 **공동체 충성도**
- “나는 누구 닮았나?”가 아니라
- “나는 공동체에서 끝까지 남는 사람인가?”를 묻는다.

### 4.2 테스트만이 아니라 **결과 분기형 경험**
- 충신이 나오면 왕을 지키는 게임
- 생존자가 나오면 유배길 생존 게임

### 4.3 외부 영화평보다 **서비스 내부 집단 심리 시각화**
- 전체 참여자 중 충신 비율
- 가장 많이 선택된 감정
- 결과 희귀도
- 게임 최고 기록

# 6. 랜딩 페이지 상세 설계

## 6.1 랜딩 목표
사용자가 3초 안에 “이거 해봐야겠다”를 느끼게 만드는 것.

## 6.2 랜딩 상단 카피
### 메인 카피 후보
- **당신은 공동체에서 끝까지 남는 사람인가요?**
- **당신은 왕을 지킬 사람인가, 먼저 떠날 사람인가**
- **진짜 충신은 많지 않습니다**

### 추천 메인 카피
**당신은 공동체에서 끝까지 남는 사람인가요?**

### 서브카피
**회사, 학교, 팀플 속 당신의 충성도를 판별합니다.**

## 6.3 랜딩 후킹 통계
랜딩 상단에 아래 3개를 노출한다.

- `총 참여자 수`
- `충신 비율`
- `오늘 가장 많이 나온 감정`

예시:
- 현재까지 **18,241명** 참여
- 진짜 충신은 **14%**
- 오늘 가장 많이 나온 감정: **여운**

## 6.4 랜딩 CTA
- 버튼 1: **충성도 테스트 시작하기**
- 버튼 2: **내가 왕을 지킬 사람인지 확인하기**

## 6.5 랜딩 중간 섹션
### 섹션 A. 감정 키워드 클라우드
서비스 내부에서 수집한 감정 선택 데이터를 시각화한다.

예:
- 여운
- 불안
- 충성
- 인간성
- 책임
- 배신
- 생존
- 침묵

### 섹션 B. 공동체 충성도 분포
막대 그래프 또는 카드형:
- 절대 충신: 12%
- 현실 충신: 28%
- 전략 생존자: 32%
- 독립 생존자: 18%
- 조용한 이탈자: 10%

### 섹션 C. 서비스 소개
짧게 설명:
> 영화가 끝난 뒤 남는 질문을 현실의 공동체로 가져왔습니다.  
> 당신은 위기 앞에서 남는 사람입니까, 떠나는 사람입니까?

# 7. 충성도 테스트 설계

## 7.1 기본 원칙
- 질문 수는 10개
- 각 문항은 4지선다
- 중립 없음
- “정치”가 아니라 “공동체 딜레마”
- 회사/학교/팀플/친구 모두 공감 가능해야 함

## 7.2 질문 데이터 구조
각 질문은 다음 필드를 가진다.

```python
{
    "id": 1,
    "text": "팀 프로젝트에서 한 사람이 큰 실수를 했습니다.",
    "options": [
        {"id": "A", "text": "내가 대신 나서서 수습한다", "score": 10},
        {"id": "B", "text": "조용히 도와주되 티는 내지 않는다", "score": 7},
        {"id": "C", "text": "상황을 지켜보다 필요하면 움직인다", "score": 4},
        {"id": "D", "text": "각자 책임이라 생각하고 내 일만 한다", "score": 1}
    ]
}
```

# 10. 희귀도 계산

## 10.1 목표
사용자에게 “내 결과가 얼마나 흔한가 / 희귀한가”를 보여준다.

## 10.2 계산 방식
DB에 저장된 전체 결과 수를 기준으로 현재 타입의 비율을 계산한다.

예:
- 전체 결과 10,000건
- 절대 충신 1,200건
→ 절대 충신 비율 12%

## 10.3 결과 노출 방식
- **상위 12%**
- **전체 참여자 중 14%만 나온 결과**
- **오늘 가장 희귀한 유형**

## 10.4 주의
초기 데이터가 적을 때는 표본 왜곡이 있으므로 다음처럼 처리:
- 전체 응답 100건 미만이면 “집계 중” 표시
- 100건 이상부터 퍼센트 노출

# 11. 서비스 내부 워드클라우드 설계

## 11.1 외부 영화평이 아닌 내부 데이터 기반으로 간다
워드클라우드는 외부 리뷰 크롤링보다 **서비스 내부 사용자 감정 데이터**를 사용한다.

이유:
- 구현이 간단하다
- 진짜 실시간 느낌을 줄 수 있다
- 참여와 연결된다
- 서비스 데이터가 곧 콘텐츠가 된다

## 11.2 수집 방식
테스트 종료 시 감정 선택 1개 저장:
- 여운
- 충성
- 배신
- 불안
- 책임
- 인간성
- 생존
- 침묵

## 11.3 표시 방식
프론트에 JSON 배열로 제공:
```json
[
  {"word": "여운", "count": 1432},
  {"word": "충성", "count": 1121},
  {"word": "불안", "count": 941}
]
```

## 11.4 갱신 주기
1차 MVP에서는 페이지 새로고침 시 갱신으로 충분하다.  
실시간 소켓은 필요 없다.

# 12. 게임 설계

## 12.1 공통 원칙
- 모바일/데스크톱 모두 가능
- 짧고 직관적
- 10초 안에 규칙 이해 가능
- 결과 문구가 밈처럼 남아야 함

---

## 12.2 게임 1: 왕을 숨겨라

### 대상
- 절대 충신
- 현실 충신

### 콘셉트
왕을 지키며 발각을 피하는 회피형 미니게임

### 게임 규칙
- 플레이어는 화면에서 이동하는 캐릭터
- 장애물을 피하며 생존 시간 증가
- 맞으면 게임 종료

### 장애물
- 관군
- 소문 말풍선
- 밀고문
- 의심의 눈

### 아이템 (MVP에서는 생략 가능)
- 은신처
- 충신 도움

### 결과 문구
- 0~15초: 왕을 너무 빨리 잃었습니다
- 16~35초: 마음은 충신, 실전은 초보
- 36~60초: 광천골의 수호자
- 61초 이상: 왕과 함께 끝까지 버틴 자

### 구현 추천
- HTML Canvas + JavaScript
- 백엔드는 기록 저장 API만 담당

---

## 12.3 게임 2: 유배길 생존 게임

### 대상
- 전략 생존자
- 독립 생존자
- 조용한 이탈자

### 콘셉트
도망/회피형 레트로 생존 게임

### 게임 규칙
- 좌우 이동 또는 드래그
- 내려오는 장애물 피하기
- 생존 시간 기록

### 장애물
- 추격자
- 굶주림 상징 아이콘
- 진창길
- 흩어진 소문

### 결과 문구
- 0~10초: 첫날 밤을 넘기지 못했습니다
- 11~30초: 위기 앞에서 흔들리는 생존자
- 31~60초: 유배길의 생존자
- 61초 이상: 끝내 살아남은 사람

# 18. API 설계

## 18.1 페이지 라우트
- `GET /` → 랜딩 페이지
- `GET /test` → 테스트 페이지
- `GET /result/{session_uuid}` → 결과 페이지
- `GET /game/loyal/{session_uuid}` → 왕을 숨겨라
- `GET /game/survival/{session_uuid}` → 유배길 생존

## 18.2 테스트 API
- `POST /api/test/submit`
  - 입력: 답변 목록 + 감정 선택
  - 처리: 점수 계산, 결과 저장
  - 출력: session_uuid, total_score, result_type

### 요청 예시
```json
{
  "answers": [
    {"question_id": 1, "selected_option": "A"},
    {"question_id": 2, "selected_option": "B"}
  ],
  "emotion_word": "여운"
}
```

### 응답 예시
```json
{
  "session_uuid": "uuid-string",
  "total_score": 82,
  "result_type": "현실 충신"
}
```

## 18.3 통계 API
- `GET /api/stats/summary`
- `GET /api/stats/emotions`
- `GET /api/stats/results`
- `GET /api/stats/game-top`

## 18.4 게임 기록 API
- `POST /api/game/record`
  - 입력: session_uuid, game_type, score
  - 출력: success
